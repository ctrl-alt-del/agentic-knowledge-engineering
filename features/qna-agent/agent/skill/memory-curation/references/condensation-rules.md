# 记忆生命周期规则

## 何时添加

| 信号 | 评分 | 说明 |
|------|------|------|
| 用户提供了知识源中不存在的新信息 | 高 | 如："我们公司用的是自建镜像仓库" |
| 发现知识源间的交叉关联 | 高 | 如：部署指南 + 错误日志指向同一配置项 |
| 用户表达了重复出现的问题模式 | 中 | 如：连续多人问同一个部署问题 |
| 对话中提取出可复用的经验 | 中 | 如："每次 X 失败都要先检查 Y" |
| 用户偏好/角色信息 | 低 | 如：用户是运维角色，偏好命令行操作 |

## 何时合并压缩（Condense）

**条件**：两个或更多条目满足以下任一条件：
1. `content` 文本的 token 重叠率 > 70%
2. 标签完全相同 + 描述同一场景的不同方面
3. 多个条目的 sources 几乎完全重叠

**操作**：
- 合并 content：保留最精炼、最完整的表述
- 合并 sources：去重后保留全部
- 合并 tags：去重后保留全部
- 置信度：取最高的
- access_count：相加
- 删除被合并的旧条目

## 何时遗忘（Forget）

**条件**（需同时满足）：
1. `stale == true`（已被取代或过期标记）
2. `access_count < 3`（很少被引用）
3. 距离 `created` 超过 `stale_threshold_days`（默认 30 天）

**例外**：`type == "gotcha"` 且带有 ⚡ 标记的条目永不遗忘。

## 何时更新

**条件**：新信息直接否定旧信息，且新信息置信度为 high：
- 如：旧记忆「API 使用 POST 方法」，新对话证实「API 已改为 PUT」
- 如：旧记忆「默认端口 8080」，新文档显示「默认端口改为 9090」

**操作**：
1. 标记旧条目 `stale: true`
2. 设置 `superseded_by: "新条目ID"`
3. 创建新条目，`confidence: "high"`

## 蒸馏触发

**自动触发**（浏览器检测）：
- 对话空闲超过 30 秒
- 浏览器标签页隐藏（用户切换标签）

**手动触发**（UI 按钮）：
- 用户点击「🧠 蒸馏」按钮
- 在蒸馏提示中，orchestrator 应审查最近的对话历史，
  识别可记录的 learning/pattern/gotcha/connection，添加到持久记忆。

## 记忆导出

**触发**：用户点击「⬇ 导出」按钮
**格式**：`persistent.json` 完整下载
**兼容性**：导出的 JSON 可直接导入其他 qna-agent 实例
